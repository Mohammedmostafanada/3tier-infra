## Network Module

### VPC

#### Key Concepts

- A VPC is an isolated virtual network inside AWS.
- Every network resource (Subnets, Route Tables, Internet Gateway, NAT Gateway, EC2) must belong to a VPC.
- The CIDR block defines the IP address range of the VPC.
- We selected `10.0.0.0/16` because it provides enough address space for multiple subnets while keeping the design simple.
- `enable_dns_support = true` enables DNS resolution inside the VPC.
- `enable_dns_hostnames = true` allows EC2 instances to receive DNS hostnames.
- The `Name` tag is displayed in the AWS Console.
- `main` is the Terraform local resource name and is only used inside the code.

#### Resource

```hcl
resource "aws_vpc" "main" {

  cidr_block = "10.0.0.0/16"

  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = {
    Name = "3tier-vpc"
  }

}
```

#### Notes

- `instance_tenancy` was omitted because its default value is `default`.
- The VPC has not been created yet because the Network Module is not called from the Root Module.



### Subnets

- A subnet is a smaller network created inside a VPC.
- Every subnet belongs to exactly one VPC using the `vpc_id` argument.
- We divided the VPC (`10.0.0.0/16`) into `/24` subnets for better organization and future scalability.
- `availability_zone` determines which Availability Zone the subnet is created in.
- A subnet is **not** public or private by itself; this is determined later by its Route Table.
- `map_public_ip_on_launch = true` automatically assigns a public IP to EC2 instances launched in the subnet.
- `map_public_ip_on_launch = false` means instances receive only private IP addresses by default.
- Public IP assignment is different from `map_customer_owned_ip_on_launch`, which is used with AWS Outposts.
- Terraform automatically detects dependencies between resources when using references such as `aws_vpc.main.id`; no `depends_on` is needed in this case.
- During `terraform plan`, values generated by AWS (such as resource IDs) appear as `(known after apply)` because they do not exist until the resources are created.


#### Notes
- Terraform builds dependencies automatically using resource references.
- Resource IDs are generated by AWS after creation and stored in the Terraform state.

### Internet Gateway

- An Internet Gateway (IGW) allows communication between a VPC and the Internet.
- An Internet Gateway is attached to a VPC, not to individual subnets.
- A VPC can have only one attached Internet Gateway.
- Creating an Internet Gateway alone does not provide Internet access.
- A Route Table must contain a route () pointing to the Internet Gateway for public Internet connectivity.
- Terraform automatically attaches the Internet Gateway to the VPC using .

#### Key Takeaways

- Internet Gateway is a VPC-level resource.
- Public and private subnets are determined by Route Tables, not by the Internet Gateway itself.
- Terraform builds dependencies automatically using resource references and may create independent resources in parallel.

### Route Tables

- A Route Table defines how traffic leaves a subnet.
- Every Route Table belongs to a VPC.
- AWS automatically creates a local route for communication inside the VPC.
- The local route does not need to be created manually.
- A public Route Table contains a default route (`0.0.0.0/0`) pointing to the Internet Gateway.
- A subnet does not use a Route Table until it is associated with it.
- Terraform uses `aws_route_table_association` to connect a subnet to a Route Table.

#### Key Takeaways

- The Internet Gateway is referenced inside the Route Table, not inside the subnet.
- A private Route Table uses a NAT Gateway instead of an Internet Gateway.
- Multiple private subnets can share the same private Route Table.
- Route Table Associations determine which Route Table each subnet uses.



---

### NAT Gateway

#### Key Concepts

- A NAT Gateway allows resources inside private subnets to access the Internet.
- Internet connections are outbound only; external systems cannot initiate connections to private instances through the NAT Gateway.
- A NAT Gateway must be created inside a public subnet.
- A NAT Gateway requires an Elastic IP because it needs a public IP address to communicate with the Internet.
- Private Route Tables use the NAT Gateway as the default route (`0.0.0.0/0`).
- The NAT Gateway depends on the Internet Gateway even though Terraform cannot infer this dependency automatically.

#### Key Takeaways

- Private EC2 instances never receive public IP addresses.
- Internet access for private instances is provided through the NAT Gateway.
- `depends_on` was required because Terraform cannot detect the relationship between the NAT Gateway and the Internet Gateway automatically.


---

### Elastic IP

#### Key Concepts

- An Elastic IP (EIP) is a static public IPv4 address provided by AWS.
- In this project, the Elastic IP is attached to the NAT Gateway, not to EC2 instances.
- Terraform automatically retrieves the Allocation ID using `aws_eip.nat.id`.

#### Key Takeaways

- The Elastic IP allows the NAT Gateway to communicate with the Internet.
- Private EC2 instances remain private while sharing the NAT Gateway's public IP for outbound traffic.
